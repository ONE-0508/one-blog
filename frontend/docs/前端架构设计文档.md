- **文档目的**：在《个人博客网站 PRD》与《前端页面设计文档》的基础上，定义前端实现层面的整体架构方案，包括：
  - 技术选型与基础约束
  - 项目目录结构与模块划分
  - 路由与布局体系
  - 状态管理与数据流设计
  - 接口访问与错误处理规范
  - 组件与样式规范
  - 开发流程与质量保障
- **适用对象**：前端开发、后端同学（对接接口）、测试同学。
- **不覆盖内容**：不详细展开接口字段、数据库结构等后端实现细节，这些应由后端技术设计文档单独定义。

## 2. 技术栈与基础约束

- **框架与基础库**
  - `React 19`：**函数组件 + Hooks** 为唯一推荐写法，不再使用 class 组件。
  - `React Router 7`：负责前端路由与页面级代码拆分。
  - `TypeScript`：全量使用 TS，禁止在业务代码中使用 `any`（除非极端情况下采用显式 `any` 并注明原因）。
  - `Vite`：开发与构建工具，统一使用 `pnpm` 作为包管理工具（如无特殊说明）。
  - `Tailwind CSS 4`：原子化样式 + 少量自定义 CSS 变量，实现明暗主题与响应式布局。
- **运行环境与目标平台**
  - 浏览器：Chrome / Safari / Firefox / Edge 近两版本；iOS Safari / Android Chrome。
  - 断点：参考 Tailwind 默认断点（`sm`、`md`、`lg`、`xl`），与页面设计文档的移动端优先策略保持一致。
- **编码基础约束**
  - 统一使用 **ES 模块**，遵循 `type: "module"`。
  - 默认启用 React 编译器 / Hook 规则，遵循项目内 ESLint 规则，不在代码中禁用 `eslint-disable`，除非经过评审。

## 3. 项目目录结构设计

> 目录以 `frontend/` 为根目录，若后续有调整，请同步更新本章节。

### 3.1 顶层目录结构

```text
frontend/
├─ public/                    # 静态资源（favicon、静态图片等）
├─ src/
│  ├─ app/                    # 应用入口与根布局
│  ├─ pages/                  # 路由页面组件（Page 级）
│  ├─ features/               # 业务模块（文章、笔记、作品、留言等）
│  ├─ components/             # 可复用的通用 UI 组件
│  ├─ layouts/                # 页面/路由级布局组件
│  ├─ routes/                 # 路由配置与懒加载封装
│  ├─ services/               # 接口与数据访问层（API 封装、请求客户端）
│  ├─ stores/                 # 全局状态管理（主题、用户、全局配置等）
│  ├─ hooks/                  # 自定义 Hooks
│  ├─ utils/                  # 工具函数（格式化、常量等）
│  ├─ types/                  # 全局类型定义（接口响应、实体类型）
│  ├─ styles/                 # 全局样式、Tailwind 配置增强
│  └─ main.tsx                # 应用入口（挂载 React）
├─ eslint.config.js           # ESLint 配置
├─ index.html                 # Vite 入口 HTML
└─ vite.config.ts             # Vite 配置
```

### 3.2 `pages/` 与 `features/` 划分

- **`pages/`**：对应路由级组件（与《前端页面设计文档》中的每个页面一一映射），主要负责：
  - 接入路由参数、查询参数。
  - 组合 `features/` 中的业务组件和 `components/` 中的通用组件。
  - 触发数据加载与状态初始化。
- **`features/`**：按业务领域划分，封装与特定领域相关的逻辑、UI 与状态，例如：

```text
src/features/
├─ posts/              # 文章相关（列表、详情、卡片、TOC 等）
│  ├─ components/
│  ├─ hooks/
│  ├─ services/        # 可选，文章特有 API 封装（不与全局 services 冲突时）
│  └─ utils/
├─ notes/              # 笔记相关
├─ works/              # 作品展示相关
├─ guestbook/          # 留言板相关
└─ stats/              # 数据统计相关
```

### 3.3 `components/` 通用组件

- 按功能或类型组织，而非业务领域，例如：

```text
src/components/
├─ layout/           # Header、Footer、Sidebar、AppShell 等
├─ navigation/       # Breadcrumb、Tab、Pagination 等
├─ data-display/     # Card、Badge、List、EmptyState、Skeleton 等
├─ form/             # Input、Textarea、Button、Select、FormField 等
├─ feedback/         # Toast、Modal、Drawer、Alert 等
└─ typography/       # Title、Text、CodeBlock 等
```

- 原则：
  - 通用组件**不直接依赖业务实体类型**（如 Post、Note 等），如需展示特定领域信息，请放在对应 `features/*/components` 下。
  - 通用组件使用 **受控/非受控 props**，不直接处理 API 调用和全局状态。

## 4. 路由与布局架构

### 4.1 路由配置结构

- 使用 `React Router 7` 的数据路由能力，在 `src/routes/` 中集中维护路由表：

```text
src/routes/
├─ index.tsx         # 根路由配置，导出 <AppRouter />
├─ routes.public.ts  # 前台（读者端）路由配置
└─ routes.error.tsx  # 通用错误与 404 页面
```

- 推荐路由配置示例（简化）：

```ts
// src/routes/routes.public.ts
export const publicRoutes = [
  { path: "/", element: <HomePage /> },
  { path: "/post/:slug", element: <PostDetailPage /> },
  { path: "/notes", element: <NotesPage /> },
  { path: "/notes/:id", element: <NoteDetailPage /> },
  { path: "/category/:name", element: <CategoryPage /> },
  { path: "/tag/:name", element: <TagPage /> },
  { path: "/archives", element: <ArchivesPage /> },
  { path: "/guestbook", element: <GuestbookPage /> },
  { path: "/works", element: <WorksPage /> },
  { path: "/works/:slug", element: <WorkDetailPage /> },
  { path: "/stats", element: <StatsPage /> },
  { path: "/about", element: <AboutPage /> },
  { path: "/links", element: <LinksPage /> },
  { path: "/search", element: <SearchPage /> },
  { path: "*", element: <NotFoundPage /> }
];
```

> 后台 `/admin` 相关路由可在后续迭代中引入单独的路由配置文件 `routes.admin.ts`，并在 `AppRouter` 中挂载。

### 4.2 布局组件

- **`AppLayout`**
  - 作为前台页面的根布局，包含：
    - `Header`（导航 + 搜索 + 主题切换）
    - `<Outlet />` 或主内容容器
    - `Footer`
  - 负责根据当前路由决定是否展示某些全局元素（例如某些特殊页可以隐藏侧边栏）。
- **`MainContent` / `Sidebar` 布局**
  - 对应 PRD 中对首页、文章详情页侧边栏的要求，在布局组件中封装两栏布局逻辑：
    - 桌面端使用两栏布局（CSS Grid / Flex）
    - 移动端自动变为单列。
- **错误与加载页面**
  - `ErrorBoundary`：捕获运行时错误，展示友好提示。
  - `LoadingScreen`：用于路由级懒加载时的占位。

## 5. 状态管理与数据流

### 5.1 状态分类

- **本地 UI 状态**（组件内）：模态框开关、表单输入、当前 Tab 等 —— 使用 `useState`、`useReducer`。
- **跨组件 UI 状态**：主题（明暗）、全局 Loading、全局 Toast 等 —— 放在 `stores/` 中，通过 Context + 自定义 Hooks 提供。
- **服务端数据状态**：文章列表、详情、笔记、作品、留言列表、统计数据等 —— 推荐使用以下策略：
  - 首版可采用自封装的 `useFetch` / `useQuery` Hook 管理加载/错误/缓存。
  - 若后续接入 React Query / SWR 等库，应在本章节更新约定。

### 5.2 全局状态设计

```text
src/stores/
├─ theme-store.ts        # 主题（light/dark/system）状态与切换逻辑
├─ app-config-store.ts   # 站点配置（站点名称、描述、社交链接等）
└─ ui-store.ts           # 全局 UI（Toast、Modal、全局 Loading 等）
```

- **主题状态（Theme Store）**
  - 来源：
    - 浏览器 `prefers-color-scheme`。
    - 用户手动选择（持久化到 localStorage）。
  - 行为：
    - 将 `dark` 类挂到 `html` 或 `body` 上，交给 Tailwind 控制样式。
- **站点配置（App Config Store）**
  - 从后端接口或静态配置拉取站点信息（名称、描述、社交链接等）。
  - 由 `AppLayout` 或顶层组件在应用初始化时获取。

### 5.3 数据获取与缓存

- 通用数据获取 Hook 示例（可根据实际实现调整）：
  - `useApiQuery`：封装 GET 请求（列表、详情）。
  - `useApiMutation`：封装 POST/PUT/DELETE 请求（留言提交等）。
- 原则：
  - **页面级组件负责触发数据加载**，将数据通过 props 传入下层组件。
  - 下层组件不直接调用 API，只接受数据与回调。
  - 对于列表与详情等常用请求，可在 `services/` 内提供专用函数（如 `fetchPosts`、`fetchPostBySlug`）。

## 6. 接口访问与错误处理规范

### 6.1 API 客户端封装

- 在 `src/services/` 下定义统一的 HTTP 客户端与 API 模块：

```text
src/services/
├─ http-client.ts      # 基于 fetch 或 axios 的基础封装
├─ posts-api.ts        # 文章相关接口
├─ notes-api.ts        # 笔记相关接口
├─ works-api.ts        # 作品相关接口
├─ guestbook-api.ts    # 留言相关接口
└─ stats-api.ts        # 统计相关接口
```

- `http-client.ts` 建议职责：
  - 统一设置基础 URL（通过环境变量 `VITE_API_BASE_URL`）。
  - 统一添加公共 Header（如 `Content-Type`）。
  - 处理基础错误（网络错误、非 2xx 状态码等），将错误转换为统一的错误对象。

### 6.2 错误处理规范

- **请求层**：
  - 所有请求返回 `Promise<Result<T>>` 的形式（可选），例如：
    - 成功：`{ ok: true, data }`
    - 失败：`{ ok: false, error: { code, message } }`
- **页面层**：
  - 在页面级 Hook 中处理加载、错误、空状态：
    - 加载中：显示 Skeleton 或 Loading。
    - 错误：展示带有重试按钮的错误提示组件。
    - 空数据：展示 `EmptyState` 组件。
- **表单提交（留言、搜索等）**：
  - 提交前：前端先做基础校验（必填、长度）。
  - 提交中：禁用按钮，显示加载状态。
  - 提交完成：
    - 成功：使用全局 Toast 提示，并按需求清空或保留用户输入。
    - 失败：根据错误码展示对应提示（如频率限制错误提示“操作太频繁，请稍后再试”）。

## 7. 组件与样式规范

### 7.1 组件分层原则

- **Page 组件（`pages/*`）**
  - 与路由一一对应。
  - 负责：
    - 解析路由参数、URL 查询参数。
    - 调用 `services/` 获取数据。
    - 将数据与回调通过 props 下发给 Feature 组件。
- **Feature 组件（`features/*/components`）**
  - 聚合与某个业务密切相关的 UI 与逻辑，如文章列表、文章详情头部、留言表单等。
  - 不直接依赖路由对象（`useParams` 等）。
- **通用 UI 组件（`components/*`）**
  - 只处理视觉与交互，不关心业务语义（例如通用按钮、卡片、模态框）。
  - 使用统一的 props 约定与样式系统。

### 7.2 命名与文件组织

- **文件命名**：
  - React 组件文件：`PascalCase.tsx`（如 `PostCard.tsx`）。
  - Hook 文件：`useXxx.ts`（如 `useTheme.ts`、`useApiQuery.ts`）。
  - 工具函数：`camelCase.ts`（如 `formatDate.ts`）。
- **组件导出方式**：
  - 单文件单默认导出 + 命名导出组合使用。
  - 对于 Feature 内部组件，可以只在 Feature index 中统一导出。

### 7.3 Tailwind 与样式策略

- 全局：
  - 使用 `@tailwind base; @tailwind components; @tailwind utilities;` 初始化。
  - 使用 CSS 变量管理主题色、背景色、边框色等，Tailwind 通过配置引用变量。
- 组件：
  - 优先使用 Tailwind 原子类构建样式。
  - 对于复杂组件（如卡片、按钮）可通过组合类 + 自定义 `className` 支持扩展。
  - 禁止在组件内散落内联样式（除非确有必要且有备注）。

## 8. TypeScript 与代码规范

### 8.1 类型定义与复用

- 所有接口返回数据、实体模型必须在 `src/types/` 中定义，例如：

```text
src/types/
├─ post.ts      # 文章相关类型
├─ note.ts      # 笔记相关类型
├─ work.ts      # 作品相关类型
├─ guestbook.ts # 留言相关类型
└─ stats.ts     # 统计相关类型
```

- 原则：
  - 不在组件中直接写临时的 `type`，通用类型放入 `types`，局部仅供组件内部的类型可以定义在文件顶部。
  - 接口类型与后端对齐，由后端技术文档产出后同步更新。

### 8.2 编码风格与 ESLint

- 严格遵循项目内 `eslint.config.js` 的规则：
  - 遵守 React Hooks 规则（`rules-of-hooks`）。
  - 避免未使用变量、未处理的 Promise 等。
  - 不通过 `eslint-disable` 粗暴屏蔽规则，如确有需要应在代码评审中说明。

### 8.3 常见约定

- 使用 `const` 优先于 `let`，尽量避免可变引用。
- 不在组件顶层之外调用 Hook。
- 组件 props 推荐使用接口 `interface XxxProps` 而不是类型别名（保持可扩展性）。
- 逻辑分支清晰，避免嵌套过深（适当拆分为小函数或子组件）。

## 9. 开发流程与协作规范

### 9.1 本地开发流程

1. 安装依赖：
   - `pnpm install`（或根据团队实际选择 npm/yarn，但需统一）。
2. 启动开发服务器：
   - `pnpm dev`，访问默认地址（如 `http://localhost:5173`）。
3. 按模块开发：
   - 根据 PRD 与页面设计文档，优先实现核心页面（首页、文章详情、笔记、留言板、作品展示）。
4. 自测：
   - 确认在 Chrome / Safari / 移动端模拟下无明显布局与交互问题。

### 9.2 提交流程

- 每次提交前：
  - 运行 `pnpm lint`，确保无 ESLint 错误。
  - 可选：运行 `pnpm build` 确认构建通过。
- 提交信息：
  - 推荐使用简洁、规范的英文或中文描述，如：
    - `feat: 实现文章详情页基础布局`
    - `fix: 修复移动端导航遮挡问题`

### 9.3 开发节奏建议

- **首版优先级**（结合 PRD 的 P0/P1）：
  - 阶段 1：搭建基础架构与路由；实现首页、文章详情、笔记列表、留言板、作品展示。
  - 阶段 2：实现分类/标签页、归档、关于页、友链页、搜索。
  - 阶段 3：完善数据统计页、RSS 入口等扩展能力。

## 10. 性能与可维护性考虑

### 10.1 性能优化

- 路由级懒加载：对于访问频率较低的页面（如统计页、关于页、友链页），使用 `React.lazy` 与 `Suspense` 实现代码分割。
- 图片与资源：
  - 使用 `loading="lazy"` 的图片懒加载策略（文章插图、作品封面等）。
  - 控制图片大小与格式，避免过大资源阻塞首屏。
- 列表渲染：
  - 使用稳定的 `key`（如 ID 或 slug），避免数组索引作为 key。

### 10.2 可维护性

- 模块边界清晰：不跨模块引用内部实现（如 `features/posts` 不直接访问 `features/notes` 的内部组件）。
- 避免“上帝组件”：当某个组件代码过长或职责过多时，及时拆分为子组件或 Hook。
- 文档同步：当架构层面有重大调整（目录变更、状态管理方式更换等），需更新本文档相应章节。

## 11. 后续扩展方向

- 后台管理端前端（`/admin`）：
  - 可在同一仓库内新建 `admin` 子应用或在当前 React 应用中增加 `admin` 路由分支。
  - 推荐复用现有通用组件与样式体系，减少重复。
- 多语言支持：
  - 若 PRD 引入多语言需求，可在 `stores/app-config-store.ts` 中增加语言状态，并引入 i18n 方案（如 `react-i18next`）。
- 插件与小组件：
  - 如“相关推荐”“最近文章”等，可在 `features/widgets` 中扩展，统一管理。
